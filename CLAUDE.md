# CLAUDE.md

This file provides guidance to Claude Code (claude.ai/code) when working with code in this repository.

## What this is

A wedding invitation website (Shikha & Amit, 17–18 Jan 2027, Jim Corbett, "Mango Bloom River Resort"). The **active project is [site/](site/)** — a React + Vite build using the "Orchid Cream" palette: a tap-to-open envelope, scratch-to-reveal hero date, per-event unlock mini-games (matka whack / dhol tap / garland tie) that open a details modal, story detail pages, dual-number WhatsApp + Google-Sheet RSVP. See [site/README.md](site/README.md) and [site/SETUP.md](site/SETUP.md) for how to develop and configure it. Everything below describes the **archived v1**, kept at the repo root untouched for reference.

## Repository layout

- [site/](site/) — the current app. React + TypeScript + Vite. Start here for any new work.
- [index.html](index.html) — archived v1: the entire old site (markup, CSS, JS all inline in one file). No longer the active target for changes.
- `shikha-amit-wedding-site.zip` — archived v1's packaged snapshot for distribution/upload. Contains `shikha-amit-wedding/index.html` (byte-identical to the root `index.html`) plus `fonts/` and `img/` assets referenced by relative paths (`fonts/*.ttf`, `img/*.jpg`). These same font/image assets were unzipped into `site/public/` for the v2 rebuild.
- `PREVIEW.png` — a rendered screenshot/preview of the old v1 page.

The top-level `index.html` has no local `fonts/` or `img/` directories next to it — those assets only exist inside the zip. To preview the page with fonts/images loading correctly, work from an unzipped copy of `shikha-amit-wedding-site.zip` (open `shikha-amit-wedding/index.html` in that extracted tree), or extract the zip's `fonts/` and `img/` folders next to `index.html`.

**After editing `index.html`, keep the copy inside the zip in sync** (or regenerate the zip) since they're meant to be identical — the zip is what actually gets deployed/uploaded. This only applies if you're intentionally touching the archived v1; new work belongs in `site/`.

## Working with the archived v1 (index.html)

- No build/lint/test tooling exists. Verify changes by opening the HTML directly in a browser.
- All CSS is in one `<style>` block using CSS custom properties defined on `:root` (`--paper`, `--ink`, `--gold`, `--rose`, `--forest`, `--teal`, etc.) — reuse these variables rather than introducing new colors.
- All JS is in one `<script>` block at the bottom, organized as small IIFEs/sections by feature: mobile nav toggle, countdown timer, scroll-reveal (`IntersectionObserver`), firefly particle animation, canvas-based "scratch to reveal" effect, and RSVP iframe embed.
- Two config values are meant to be edited directly in the script rather than templated:
  - `TARGET` — the countdown target date/time.
  - `RSVP_FORM_EMBED_URL` — paste a Google Form embed URL here to activate the RSVP iframe (see the placeholder text in the `#rsvp` section).
- Respects `prefers-reduced-motion`: fireflies, scroll animations, and the scratch-reveal canvas all have reduced-motion fallbacks — preserve this when touching animation code.
- Fonts are self-hosted via `@font-face` (`fonts/*.ttf`) — Vibes (script), Corm/CormI (serif body, variable font), Jost (UI/labels), Deva (Devanagari blessing text). No CDN/webfont dependency.

## Working with site/ (active project)

- Commands (run from `site/`): `npm run dev`, `npm run build` (`tsc -b && vite build`), `npm run preview`, `npm run lint` (oxlint).
- **Config/placeholders**: `src/config.ts` reads `VITE_*` env vars (dual WhatsApp numbers, RSVP deadline, venue maps link, Apps Script URL) with safe fallbacks — the app builds and runs with none of them set. `venueMapsLink` and `rsvpDeadline` now default to *real* confirmed values (not placeholders) — only the WhatsApp numbers and Apps Script URL are still genuinely unset until the couple supplies them. Components check `isPlaceholder.*` and show a "not configured yet" state rather than breaking. Don't hardcode these values elsewhere — always go through `config.ts`.
- **Dual WhatsApp routing**: `config.whatsappBride`/`config.whatsappGroom` (`VITE_WHATSAPP_BRIDE`/`VITE_WHATSAPP_GROOM`), not a single number. `lib/whatsapp.ts`'s `pickWhatsappNumber(teamPick)` routes to the matching side, falling back to the bride's number when no team is picked or that side isn't configured — `RsvpForm` calls this at submit time rather than reading `config.whatsappBride` directly. `isPlaceholder.whatsapp` is only true when *both* are unset.
- **RSVP data flow**: `TeamPick` and `GettingThere` write into `RsvpContext` (`src/context/RsvpContext.tsx`); `RsvpForm` also owns local state for `attending`/`celebrations`/`contact`/`songRequest` (see `RsvpPayload` in `lib/whatsapp.ts`), plus reads/writes `wantsTrainCoach` from context directly (the form has its own "Coming on the wedding train from Patna?" toggle now, in addition to the one in `GettingThere` — both write the same context field, so either one flips it for the other). Fires both `lib/whatsapp.ts` (prefilled `wa.me` link) and `lib/sheet.ts` (fire-and-forget POST to Apps Script, `mode:'no-cors'`, catches its own errors) from one handler at submit time — keep both firing together. `CELEBRATION_LABEL` (exported from `lib/whatsapp.ts`) is still the event-id → label map used by `EventCard`'s "Open the …" button text, but `RsvpForm` no longer reuses it for its chips — it has its own local `CELEBRATIONS` constant (Haldi/Mehendi/Sangeet/The Wedding, 4 items) because the RSVP form intentionally splits Mehendi and Sangeet into separate chips even though they share one `EventCard`/modal. Don't try to unify these two lists — they answer different questions (which event-card unlocks vs. which specific functions a guest is attending). Two things read `teamPick` from context to reflect the pick site-wide: `PetalsLayer` swaps its falling-petal palette, and `TeamHueOverlay.tsx` (`z-index:60`, `pointer-events:none`) washes the *entire* viewport in a matching tint.
- **Dhol tap stats**: `TapTheDhol.tsx` (the full-featured standalone game in `DholSection`, further down the page) tracks three separate numbers, don't collapse them — `taps` (this session only, resets on reload), `best` (highest single-session count, `localStorage` key `amishi-dhol-best`), and `total` (lifetime cumulative count across every visit, `localStorage` key `amishi-dhol-total`). There's a second, deliberately simpler `MiniDholTap.tsx` used inline in the Mehendi & Sangeet `EventCard` as its unlock mechanic (5 taps, no stats, no shake) — the two are not meant to be merged; the mini one just gates the modal, the full one is the "keep playing" game.
- **`ScratchReveal`** (`src/components/ScratchReveal.tsx`) is the reusable canvas "scratch to reveal" component — used only by `Hero` (the wedding-date box). Event cards no longer use it (see below); don't re-add it there without checking whether that's actually wanted again.
- **Per-event unlock mechanics** (`src/components/games/`): each `EventCard` shows a compact teaser (tag/title/when — no `lede` on the card itself, that moved into the modal) plus a *different* mini-game per event — `MatkaWhack` (Haldi, 3 clicks to crack/tip the pot, sound via `src/lib/potSound.ts`'s `playPotKnock()`/`playPotBreak()`), `MiniDholTap` (Mehendi & Sangeet, 5 taps, sound via `dholSound.ts`'s `playDholHit()`), `GarlandTie` (Wedding, pointer-drag one garland end into the other, snap-to-tie). All three render a row of `.mechanic-dots` (shared utility in `global.css`) tracking hit/step progress. Completing the game reveals a "Open the …" button that opens `Modal` with the full event details (photo, `dl` rows, quirk, calendar buttons, and — Haldi only — `TeamPick`). Don't put event detail content back on the page outside the modal; that was a deliberate brief requirement (every event needs a distinct interaction, not identical scratch cards). **The mechanic itself stays mounted after `onUnlock`** — `EventCard.tsx` does not conditionally hide it once `unlocked` is true, so the finished state (cracked pot, filled dots, tied garland) keeps showing next to the open button instead of disappearing; don't reintroduce a `{!unlocked && <Mechanic/>}` gate. The three cards render side by side via `.weekend-grid` (`WeekendEvents.css`, `grid-template-columns:repeat(3, 1fr)`, collapsing to one column under 820px) rather than a stacked full-width list — each card has its own colored background gradient (`.event-haldi`/`.event-mehendi-sangeet`/`.event-wedding` in `EventCard.css`) and colored pill open-button.
- **`Modal`** (`src/components/Modal.tsx`) is portalled to `document.body` via `createPortal` — deliberately, so its `position:fixed` backdrop can never be broken by a future ancestor gaining a `transform`/`filter` (which would make that ancestor the containing block instead of the viewport). Because of the portal, its content is **not** a DOM descendant of `.event` — shared typography rules in `EventCard.css` are dual-scoped (`.event h3, .modal-panel h3` etc.) rather than just `.event h3`; keep that pattern for any new shared style between the page and modal content. Closes on Escape, backdrop click, or the close button; exit animation is a fast tween (not spring) so it doesn't linger — a spring-based exit here once took ~1s to actually unmount, which looked like a stuck/broken close.
- **CSS**: one plain `.css` file per component, colocated and imported directly (no CSS Modules). Shared tokens live in `src/styles/theme.css` — the variable *names* (`--rose`, `--forest`, `--teal`, `--gold`, `--paper`, `--cream`, `--ink`, `--line`, plus `--sage`/`--purple` as legacy aliases) are kept as semantic aliases onto the "Orchid Cream" palette values on purpose, so don't rename them without checking every consumer. `--rose` is its own hue (not a `--pink` alias) — headings use it, backgrounds/petals use `--pink`. New tokens: `--orchid` (primary anchor, replaces the old `--sage`-as-green anchor), plus a `--dusk`/`--moon`/`--glow` twilight set used only by the Footer. Section-wise accents (Story=lilac, Haldi=orange, Mehendi=green, Wedding=turquoise, Getting There=blue) are applied via a handful of section-scoped overrides, not a full per-section variable system — see the "Per-event accent" block in `EventCard.css` and `.section-lilac` in `StoryTimeline.css` for the pattern.
- **`WaveDivider`** (`src/components/WaveDivider.tsx`) — a full-width SVG wave, `from`/`to` color props, dropped between sections in `HomePage.tsx` at genuine background-color transitions (paper↔cream↔dusk) — don't add one between two sections that already share a background, it won't read as anything. `decor/Toran.tsx` is a similar one-off decorative flourish (marigold string) at the top of the Weekend section; its flower positions are computed from the *same* quadratic-bezier formula as the string's SVG path so they land exactly on the curve — if you change the path, update `P0`/`P1`/`P2` to match.
- **Anchor-scroll / fixed-nav gotcha**: `html{scroll-padding-top:80px}` in `global.css` exists because `scrollIntoView()`/anchor jumps were landing interactive elements (e.g. the mini dhol tap button) right under the fixed `Nav` bar, where they were visible but their top ~70px was unclickable (nav intercepts pointer events). Any new anchor target or programmatic scroll relies on this — don't remove it without re-testing elements that sit near the top of a section.
- **Ambient layers** (`src/components/ambient/`: `FirefliesLayer`, `PetalsLayer`, `DiyasLayer`) all gate on `useReducedMotion()` (`src/lib/reducedMotion.ts`) and render nothing when reduced motion is on — preserve this on any new animated component.
- **Scroll animation**: `Reveal.tsx` (framer-motion `whileInView`, spring easing, `variant="pop"` default / `"soft"` for reading-heavy content like story paragraphs) is the scroll-pop primitive used throughout — reuse it instead of a one-off `IntersectionObserver`. It's deliberately bidirectional (`viewport={{once:false}}`): elements pop in on the way down and pop back out to their `initial` state on the way up (or if scrolled past quickly in either direction) — don't add `once:true` back without a specific reason, that was a conscious choice, not an oversight. `Hero.tsx` additionally uses framer-motion's `useScroll`/`useTransform` for a parallax effect on the hills SVG and hero content as you scroll past it; both skip entirely under reduced motion.
- **`EventCard`'s staggered reveal now lives inside the modal**, not the page. The on-page teaser (tag/title/when/lede + the unlock mechanic) is a single centered column, each piece its own `<Reveal>`. Once opened, the modal's photo/title/`dl` rows/quirk/actions are *also* individually wrapped in `<Reveal>`, staggered by `ROW_STAGGER` (0.1s) via the `delay` prop, so the content cascades in fresh every time the modal opens — this reuses the same primitive rather than inventing modal-specific entrance animation. (Earlier versions had this cascade directly on the page with the photo last in DOM order for a scroll-position reason that no longer applies now that the photo lives in the modal — don't reintroduce that if refactoring, the modal makes it moot.)
- **Dhol game feedback** (`games/TapTheDhol.tsx`): `src/lib/dholSound.ts` synthesizes a percussive hit via the Web Audio API (oscillator pitch-sweep + filtered noise burst, no audio assets) and cycles through a few pitch/decay presets so consecutive taps don't sound identical. `src/lib/screenShake.ts` toggles a `.screen-shake` class on `#root` (keyframes in `global.css`, up to ~22px translate / ~1.8deg rotate — this was deliberately made large/violent after an earlier subtler version read as too weak, don't quietly shrink it back down) when `TapTheDhol` detects `FRENZY_TAPS` (4) taps within `FRENZY_WINDOW_MS` (650ms) — it force-reflows before re-adding the class so it can retrigger mid-shake. `triggerScreenShake()` self-guards on `prefers-reduced-motion`, so callers don't need to check it themselves. The class-removal `setTimeout` in `screenShake.ts` must stay ≥ the `screen-shake` `animation-duration` in `global.css` (currently .55s / 570ms) or the class gets yanked before the animation finishes playing.
- **Routing**: React Router (`BrowserRouter` in `main.tsx`). `App.tsx` owns `<Routes>` (`/` → `HomePage`, `/story/:slug` → `StoryDetailPage`) plus a `ScrollToTop` helper that scrolls to a hash target (e.g. `/#story`) or the page top on every navigation. The envelope-opened (`entered`) flag is deliberately owned by `App`, not `HomePage` — `HomePage` remounts on every route change (e.g. navigating back from a story page), so state that must survive that has to live above the `<Routes>`.
- **Envelope open lifecycle** (`EnvelopeOpener.tsx` + `HomePage.tsx`): this is the one place with a non-obvious React-batching gotcha. `HomePage` captures `useState(entered)` once at mount into `skipEnvelope` — deliberately *not* a live re-render on every `entered` change. Reasoning: on the very first open, clicking the envelope flips `entered` true in `App` in the same batch as the component's own internal `opened` state; if the parent instead conditionally unmounted `<EnvelopeOpener>` on `entered` becoming true, React 18 batching would rip it out of the tree before its own CSS fade-out (`.envelope-gate.opened`, ~1.8s) ever painted. So `EnvelopeOpener` always stays mounted for a `HomePage` instance that started with `entered=false`, and unmounts *itself* (internal `hidden` state, `setTimeout`) once its animation finishes. Only a `HomePage` instance that was already `entered` at mount (e.g. remounted by navigating back from a story page) skips rendering it at all. Don't "simplify" this to `{!entered && <EnvelopeOpener/>}` — that's the bug this avoids.
- **Page frame**: `PageFrame.tsx` is a fixed, `pointer-events:none`, low-`z-index` (`5`, below `Nav`'s `50` and `PetalsLayer`'s `40` on purpose so the nav bar visually masks the seam where the frame's top edge would otherwise cut across it) gold double-border with corner accents, mounted once in `App.tsx` above `<Routes>` so it's present on every route. Purely decorative — it does not affect layout/padding of content underneath.
- **Fixed-overlay z-index order** (back to front): `PageFrame` (5) → `DiyasLayer`/section-local ambients (1–2, inside their own section's stacking context) → `PetalsLayer` (40) → `Nav` (50) → `TeamHueOverlay` (60, deliberately above `Nav` so the tint washes the nav bar too) → `Modal` backdrop (90) → `EnvelopeOpener` (100, only present pre-open). Keep new full-viewport overlays consistent with this ordering rather than picking an arbitrary z-index.
- **Story detail pages**: `src/data/story.ts` holds a `slug` and `details[]` (expandable narrative paragraphs) per chapter, in addition to the short `note` shown on the homepage timeline. `StoryTimeline.tsx` links each stop to `/story/:slug`; `StoryDetailPage.tsx` renders the full chapter with prev/next pager and a link back to `/#story`. The `details[]` copy is placeholder text in the couple's voice — flagged in the file for the couple to personalize with real specifics before wide distribution, not meant to stay as-is.
- The Apps Script side (`apps-script/Code.gs`) is not part of the Vite build — it's deployed manually by the couple in their own Google account (see `SETUP.md`). Don't expect `npm run build` to touch it.
- Deploy target is Vercel with **Root Directory = `site`** (the repo root is not the app root). `vercel.json` has the SPA rewrite (`/(.*) → /index.html`) required for React Router's client-side routes (e.g. `/story/chennai-2021`) to resolve on a hard refresh or direct link — if you switch hosts, replicate that rewrite rule there too.
